 **본 프로젝트는 1인 개발 프로젝트입니다.

 저는 채팅 어플리케이션을 통해 사람들과 소통할때, 텍스트 외 이모티콘과 부가 표현들을 통해서만 문자 너머에 담긴 상대의 의도와 감정을 알수있었습니다.

### 🔴 다대다 참조 문제
  - 테이블 구조 설계 간에 다대다 문제 (many to many) 문제 chatroom과 user 에서의 문제.

    user는 다수의 chatroom을 가질수있어야하고 chatroom 역시 다수의 user와 엮여야 합니다. 문제는 chatroom을 불러왔을때 chatroom은 user를 참조하고 user는 chatroom을 참조하는 무한 굴레(순환참조)에 빠지게 되는 점입니다.

    또한 ManytoMany 어노테이션으로 참조 관계 설정시 중간 테이블이 자동 생성됩니다. 이 테이블 내부는 제어도 불가합니다. 이를 해결하기 위해 중간 조인테이블인 RoomInUser테이블을 생성했고 user는 RoomInUser테이블을 통해 본인이 속한 채팅방이 어느 곳인지 명확히 확인할 수 있게 되었습니다.

    허나 이는 근본적인 순환참조의 해결방식은 아니었습니다.  User -> RoomInUser -> Chatroom -> RoomInUser -> User 와 같은 식으로 양방향 참조는 결국 서로를 참조하려들었습니다. 실행중인 어플리케이션에 콘솔로그에서 받아온 json object는 user가 무한히 뻗어나가있었습니다.

    이를 해결하기위해 DTO를 생성 UserDTO, RoomDTO 를 생성했고 테이블구조인 양방향 관계와는 다르게 roomDTO객체만 userDTO를 참조하도록 만들었습니다.

    예시로 저는 user가 자기가 속한 room을 찾을 때에 user를 인자로 전달후 대조하여 roomInUser에서 chatroom의 정보를 찾아오도록 했고 roomInUser(채팅방에 속한 유저들)는 userDTOs 라는 리스트 타입으로 변환습니다. 그리고 chatroom의 정보를 토대로 RoomDTO에 앞서 만든 userDTOS를 넣어 해당 roomDTO를 반환했습니다. 

    순환참조 문제는 이렇게 해결하였습니다.

### 🔴 동시성 문제 
  - 동시가입 1000명 시나리오를 테스트 해봤습니다. 이유는 1000명이 동시에 가입시에 응답속도가 얼마나 걸리는지 등등 궁금했습니다.
    시나리오는 이렇습니다.
    동일한 회원정보를 Jmeter를 이용해  /auth/register로 1000개의 요청을 보냅니다.
    
![스크린샷 2025-01-06 185414](https://github.com/user-attachments/assets/8caa90f5-52c4-41ab-be03-6d61a5c9dd90)



![스크린샷 2025-01-07 171024](https://github.com/user-attachments/assets/c43f8f25-3a9c-4621-8204-ed8deac2b10a)
 - 결과:
   http_request_seconds_sum : 모든 요청의 응답시간을 합한값입니다.
   그래서 697/1000 = 0.697 이기에 1개의 요청에 0.697초가 소요되었다는 얘기입니다.
   허나 중요한점은 300개 이상의 동일한 행이 삽입 됐다는 것이 중요한 문제 입니다.

   
  #### 🔵 방법1
   1. **Synchronized키워드로 회원가입 블럭 감싸기** : 한번에 하나의 쓰레드만 해당 블럭에 접근가능하도록 해보았습니다. 이후 Jmeter를 이용해 다시 1000개의 요청을 보내봤습니다.
      - 결과 : 요청의 응답시간은 동일, 동일 행 삽입수는 현저히 줄었으나 여전히 100개 이상의 동일한 행 삽입
     


  #### 🔵 방법2 
   1. **Redis 분산 Lock 적용** : Redis를 학습하며 배운 SETNX 연산을 활용하여 회원가입 요청이 들어올때 이메일 값을 이용하여 키를 생성 -> 키가 존재하면 true 아니면 false를 반환하게 하여 true일 경우에만 회원가입 로직으로 접근 가능.
      - 결과 :  동일 행 삽입없이 하나의 행만 삽입되는걸 확인했습니다. 이렇게 동시성 문제는 해결 했습니다.

### 🔴 실시간 채팅 
 - 실시간 채팅기능 구현에 있어서 단순 http 통신으로 메세지를 주고 받는 것은 거의 불가능에 가까웠습니다. http 는 statless하다는 특성 덕분에 클라이언트와 서버간의 연결을 유지하지 않았기에 지속적인 연결을 성립시키기 위한 기술이 필요했습니다.

   따라서 SockJs와 STOMP를 사용하여 메세지 전송시 엔드포인트 설정-> 해당 엔드포인트 구독 대상자에게 전송 하는 식으로 구현하고자 시도했습니다.

   하지만 연결을 수립하는 부분에서부터 문제가 발생했습니다. SockJs는 각 클라이언트 별로 고유의 세션을 생성하는데, 이 부분에서 세션의 정보가 명확하게 설정되지 않았기에 메세지를 보내는 주체와 구독하는 대상자가

   명확하지 않아 메세지 주고 받음 자체가 불가능 했습니다. 따라서 저는 WebSocket 원시 API를 이용했했습니다.

  #### 🔵 흐름
   1. **유저 로그인** : 클라이언트에서 로그인 후 WebSocket 연결을 시도 , 토큰 인터셉터에 전달.

   2. **인터셉터 처리** : 인터셉터에서 토큰을 파싱하여 이메일을 추출하고, 이를 **Map 객체**에 저장.
      
   3. **WebSocket 연결** : HTTP업그레이드 요청 , 101 응답확인 , 연결 성공
   
   4. **WsSessionManager** : 저장된 이메일을 이용해 세션을 생성.
   
   5. **채팅방 입장** : 채팅방에 입장할 때, 해당 세션을 **roomSession**에 주입.
   
   6. **메시지 전송** : 메시지를 전송할 때, 전송 대상의 이메일 값을 이용하여 세션 정보를 확인.  대상 세션이 존재하면 그 세션에 메시지를 전송.
   
   7. **메시지 수신** : 메시지를 수신하고 화면에 표시.
   
   8. **로그아웃** : 로그아웃 시, 해당 세션과 **roomSession**에 저장된 세션을 제거하고 WebSocket 연결 종료.

### 🔴 메세지 타입별 액션
 - 채팅시스템은 잘 작동했지만 반응성이 좋지는 못했습니다. 모든 메세지는 핸들러를 거쳐서 바로 상대에게 보내지는데, 이 때 콘솔에서만 메세지 수신 확인이 가능하였습니다.

   첫 채팅을 할 경우 채팅방이 새롭게 생성되기 때문에 상대가 채팅을 받기 위해서는 필수적으로 UI를 업데이트 해야했습니다.

   이러한 문제 해결과 더불어 읽음 확인 기능을 더하고 싶었습니다. 이를 구현하기위해 메세지 전송시 메세지 json객체에 type필드를 작성했고고 newMsg 인지 msg인지 enterRoom(채팅방 입장) 인지 closeRoom(채팅방 닫음) 4가지 타입을 임의로 설정하여

   타입별로 클라이언트 측에 알려주는 방식으로 해결했습니다.

  #### 🔵 흐름
   1. **일반 메세지** : 메세지의 내용과 받게될 대상을 클라이언트 측에서 핸들러로 전달 -> 핸들러 측에서 전달 대상 세션이 존재하면 전송 아니면 저장
   
   2. **신규 메세지** : 클라이언트 측은 메세지 내용과 받게될 대상만 핸들러로 전달 -> 핸들러에서 둘과의 채팅방 유무 검증후 없다면 json오브젝트에 type:"newMsg" 명시 -> 클라이언트 측 코드에서도 json 의 type필드가 "newMsg" 일경우 비동기 방식으로 정의된  roomList api 재호출 -> 화면 변경사항 반영
   
   3. **채팅방 입장** : 클라이언트 측에서 채팅방 조회시 핸들러로 json객체 전달 type:"enterRoom" -> 핸들러에서 json객체의 type이 enterRoom이라면 채팅방 상대에게 json객체 전달 doesSheInRoom :"boolean" true라면 Observer함수(MutationObserver) 호출과 OPENAI API로 메세지 내용 감정 분석 요청-> 입장시 상대에게 받게될 모든 메세지 밑에 읽음 표시와 감정분석 결과 고지.
   
   4. **채팅방 닫기** : 클라이언트 측 채팅방 닫기시 핸들러로 type:"closeRoom" json 메세지 전달 -> 핸들러에서 타입 확인후 상대 세션에게 doesSheInRoom :"false" 메세지 전달 -> MutatationObserver 종료 및 OPENAI API 사용 중지

### 🔴 User객체 삭제시 무결성 문제(회원 탈퇴)
  - 회원 탈퇴시에 해당 행을 완전히 삭제할 때 무결성 문제가 일어났습니다. User객체 삭제시에 참조 관계에 있는 message,roomInUser,friend 가 영향을 받으면서 탈퇴한 사용자와의 친구관계, 채팅 내역 등이 Orphan Records(고아 레코드) 가 되면서

    무결성 문제와 외래키 제약 위반 문제가 발생합니다. 예상 대로 조회시에 500 INTERNAL SERVER ERROR 가 발생했습니다.
  
  #### 🔵 해결
   1. **회원 탈퇴** : 기존 행 완전 삭제 로직을 soft delete 방식으로 변경했습니다. User 테이블에 ENUM 필드 생성후 타입으로는 ACTIVE,INACTIVE,DELETED로 설정해두었습니다.
                       관리자가 해당 User객체 삭제시 enum 필드는 DELETED-{phone}상태로 변경 ->  삭제 전 Redis에 키는 phone으로 찾을수있게 해두고  해당 user 정보 들을 저장.
  
   3. **회원 복구** : 회원 탈퇴시 탈퇴 버튼이 복구 버튼으로 바뀌게 되며 복구시 ENUM필드는 INACTIVE상태가 됩니다. 이전 계정정보를 Redis에서 불러와 User객체 update.
  
   4. **최종 복구** : 복구된 유저 로그인시에(enum필드가 INACTIVE일시) 비밀번호 재설정을 위한 페이지로 이동 -> JavaMailSender를 이용하여 임시 비밀번호 생성후 이를 smtp서버를 이용하여 복구된 계정의 email로 전송 -> user는 메일을 확인후 임시비밀번호 입력후 최종적으로 enum필드가 ACTIVE로 바뀌게되고 마무리됩니다.

### 🔴 RoomInUser객체 삭제시 무결성 문제(채팅방 나가기)
 - user가 해당 채팅방을 나가기하는 기능또한 아직 구현하지 않았지만 가정을 위해 db에서 임의로 RoomInUser를 삭제 해봤습니다. 이때 앞서 서술한 User객체 삭제와 동일한 문제가 생겼습니다.
   chatroom을 기준으로 채팅방의 메세지를 또 user를 기준으로 조회하기 때문에 상대가 채팅방을 나갔을 때 고아 레코드(Orphan Record)가  존재하기에 무결성 오류가 발생했습니다. 상대가 채팅방을 나가도 조회 가능도록 만들고 싶어 또 ⚡ **Redis**를 사용했습니다.

  #### 🔵 해결
   1. **채팅방 생성** : 채팅방 생성시점에 Redis에 해당 채팅방에 해당 채팅방의 연결 대상들을 UserDTO로 변환후 Chatroom Id를 키로 UserDTO들을 json형식으로 밸류로 설정해주었습니다.
    
   3. **채팅방 입장** : 채팅방 입장시 클라이언트측에서 roomId를 전달받고⚡ **Redis**에서 해당 채팅방의 정보를 조회 -> 상대가 나갔든 말든 redis의 밸류에 저장된 userDTO객체는 존재하기에 대상되는 User조회가능
    
   4. **최종**     : 채팅방을 나갔을때 roomInUser 삭제 -> 메세지 전송처리시 핸들러에서 해당 user가 실제로 RommInUser객체에 연결되어있는지 확인후 없다면 다시 추가후 메세지 전송. 즉 , 상대가 나갔어도 내가 메세지를 다시 보내게 된다면 상대는 다시 채팅방에 결속됨.

### 🔴 JWT방식 채택후 로그인 확인 
 - 기존 session기반 로그인 방식의 문제점을 보완하고자 jwt방식을 채택해보고자 했습니다.
 
   우선 토큰의 서명 및 검증을 위해 시크릿 키를 32바이트(256비트) 형식으로 생성해주기위해 프로젝트 내부에 별도로 keygen.class 를 만들어 SecureRandom클래스를 이용하여 보안적으로 안전한 난수를 new byte[32] 로 생성해 줬습니다.

   이후 해당 난수는 256비트이기에 토큰설정클래스(JwtUtil)안의 생성 메서드에서 hs256 암호 알고리즘을 사용했고 클레임과 서브젝트에 로그인 할 사용자의 email값을 인자로 설정했습니다.

   이후 클라이언트 측에서 로그인 성공시에 해당 토큰을 발급받고 클라이언트 측 로컬 스토리지에 저장했습니다. 클라이언트측에서 해당 토큰을 불러와서 로그인 검증만 하면되는데 http는 statless하단 특징덕에 모든 로그인이 필요한 api에서 해당 토큰을 불러와야만 하는 문제가 발생했습니다.

   ### 🔵 해결
   1. **로컬 스토리지 불러오기** : 토큰은 로컬스토리지에 저장되어있기에 불러와서 파싱만 해주면됩니다. 따라서 저는 checkLogin이라는 함수를 작성했습니다. 함수에서는 비동기 방식으로 js의 내장 비동기함수인 fetch함수로 작성해주었고 서버에서의 응답값을 제대로 불러오기위해 await fetch함수를 사용했습니다.

      서버에서 토큰이 유효한지를 검증 - > 유효하지 않다면 refToken을 이용하여 재발급 - > 토큰 클레임 추출 메서드 호출 -> 추출된 사용자 정보 클라이언트 측에 json형식으로 전달
      -> 클라이언트는 해당 정보를 토대로  return { loggedIn: true, userEmail: data.userEmail } 객체 반환 -> 이후 로그인 검증이 필요한 모든 api,사용자 지정 함수에서 await checkLogin 호출 -> 반환받은 객체에 저장된 값을 통해 통과 혹은 거부

   3. **단일 책임 원칙(SRP, Single Responsibility Principle)** : 단일 책임 원칙은 서버에서도 클라이언트에서도 당연시 적용되어야하는 규칙이기에 클라이언트 측에서도 하나의 api, 하나의 함수에서 하나의 역할만 하도록 하기위해 ui업데이트 및 기타 기능들을
      따로 정의해두었습니다.

                       이후 https:\\elastic-ip:8443으로 접속하니 접속이 되었습니다.

   ### 🔴 개선할 포인트
    1. **포트 리다이렉팅** : 사용자가 8080 http로 접속하여도 https 보안연결을 유도하도록 TomCatConfig.class 를 작성후 8080으로 접속되면 자동으로 8443으로 리다이렉팅되게 작성했지만 어떤 이유에서인지 되지가 않았습니다.

### 🔴 SSL 인증서 적용후 WebSocket 연결 실패
- ssl인증서 발급후 적용후에 어플리케이션 실행 -> 로그인후 WebSocket연결시 바로 연결이 업그레이드 되지 않았습니다.

  유추할수 있었던 문제점으로 WebSocket연결은 먼저 http 프로토콜로 먼저 연결요청을 보내는데 이제 https로 변경되었으니 해당 연결 url이 유효하지않다라는 생각이었습니다.

  ### 🔵 해결
   1. **웹소켓 연결 url 변경** : 기존 웹소켓 연결요청을 보낼 때 클라이언트 측에서 ws://ec2-ip:8080/ws-login?Authorization={accessToken} 으로 연결요청을 보냄
                                 ws-login 경로를 등록 -> 인터셉터 적용 -> 인터셉터에서 토큰 유효 검증후 연결 .. 이 흐름 자체는 문제가 없어보이지만 https로 바뀔때는 ws://...가 아니라
                                 wss://.... 로 바꿔줘야한다고 합니다. 그래서 해당 사항 변경후 재연결 시도해보니 성공.
